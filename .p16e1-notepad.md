## What this slice does

Closes the two P16-E audit blockers GPT raised on `00eb8e6` in a single,
tightly-scoped follow-up. No new surfaces, no new routes ‚Äî just tightening
the authorisation on the one destructive path on the operator memory page,
plus fixing the one sentence of copy that promised a workflow that
doesn't exist.

### Blocker 1 ‚Äî "Hard delete was available to any team member."

**What GPT flagged:** `src/app/memories/page.tsx:82-94` (pre-P16-E.1)
only re-checked team membership via `teamIdsForUser(me.id)` before the
delete call. A user whose global role is `viewer` but who is a member
of the team could still hard-delete durable memory. Precedent
`src/lib/ai/tools/send_campaign.ts:100-109` gates comparable
destructive/write flows at `editor+`. GPT's call: "Shipping hard delete
on membership alone is not acceptable."

**What I shipped:**

1. **Pure decision helper** ‚Äî `src/lib/memory/admin-auth.ts` defines
   `decideMemoryDeleteAuth({ isEditor, isAdmin, teamId, memberTeamIds })`
   returning `{ ok: true } | { ok: false, reason: "not_editor" | "not_member" }`.
   The decision tree is:
   - `!isEditor` ‚Üí reject `not_editor` (the new gate ‚Äî this is the
     load-bearing line that closes the audit).
   - `isAdmin` ‚Üí allow (cross-team delete mirrors the P16-E
     cross-team READ posture).
   - `isEditor && !isAdmin` ‚Üí require `memberTeamIds.includes(teamId)`.

   The helper is pure-boolean-in, object-out ‚Äî no Prisma, no User type
   leakage into tests. It lives as a separate module (not inside
   `admin-query.ts` or `ui.ts`) because its concern is orthogonal: those
   modules shape queries / display, this one gates permissions.

2. **Server Action rewrite** ‚Äî `deleteAction` in
   `src/app/memories/page.tsx` now computes the three booleans + the
   membership list, calls the helper, and maps the reject reason onto
   distinct flash copy ("Editor role required to remove durable memory."
   vs "Not authorised for that team."). The computation always runs
   `teamIdsForUser` even for admins so the input is complete; the
   helper ignores it on the admin branch. Request-scoped caching
   via `React.cache` means this is free on admin requests that also
   need the list elsewhere in the render.

3. **Render gate** ‚Äî the per-row `<form action={deleteAction}>` block
   is now conditional on `isEditor` (computed at the page top). Viewers
   see the row, provenance, body, updated-at ‚Äî everything ‚Äî but no
   Remove button. A crafted POST from a viewer is still refused
   server-side by the same helper; the render gate is pure cosmetics
   so viewers aren't confused by a button that will bounce.

4. **Header copy is role-aware** ‚Äî the last sentence of the page
   intro used to read "Remove anything that looks stale or wrong."
   unconditionally, which was confusing for viewers (no Remove button
   to match). It's now conditional: editors get the imperative, viewers
   get "An editor can remove entries that look stale or wrong."

### Blocker 2 ‚Äî "Empty-state promises a chat write path that doesn't exist."

**What GPT flagged:** `src/app/memories/page.tsx:179-180` said operators
could teach durable facts "through chat", but the shipped chat path
(`src/app/api/chat/route.ts:225-247`) only RECALLS memory. The write
seam `createMemoryForTeam` at `src/lib/memory/server.ts:69-92` has no
live callsite under `src/`. GPT's call: "tighten the copy‚Ä¶ or ship the
real write path first."

**What I shipped:**

- New empty-state text: *"Durable facts, preferences, and rules saved
  for each team will appear here once they're created."* No claim about
  HOW memory gets saved ‚Äî honest about the current state while leaving
  room for a future write path without needing a second copy change.
- The "Not in this slice" comment block at the top of the page now
  explicitly documents that the write path is deferred and that this
  page is read+delete only until it lands. The earlier wording
  ("Edit ‚Äî Delete + re-create via the chat write seam covers the
  correction case") was also misleading for the same reason and has
  been corrected.

I chose to tighten the copy rather than ship the write path. Shipping
a memory write seam is a bigger product decision (trigger surface,
validator coverage, team-attribution policy) and inflating P16-E.1
to cover it would defeat the "close the blockers fast" discipline.

## What's pinned by tests

`tests/unit/memory-admin-auth.test.ts` ‚Äî 11 pins:

- **Viewer rejected "not_editor"** (with membership in the team)
- **Viewer rejected "not_editor"** (without membership) ‚Äî proves role check wins
- **Admin allowed cross-team** (empty membership list)
- **Admin allowed** (populated list NOT containing teamId)
- **Editor in team allowed**
- **Editor NOT in team rejected "not_member"**
- **Editor with empty membership rejected "not_member"**
- **Editor with empty teamId rejected "not_member"** ‚Äî upstream-bug defence
- **Editor with non-array memberTeamIds rejected "not_member"** ‚Äî shape defence
- **Malformed input (null/undefined) rejected "not_editor"** ‚Äî fail-closed
- **Role check precedes membership check** ‚Äî explicit ordering pin (a future refactor that flipped the branches would cause viewer-on-team to get "not_member" instead of "not_editor", fragmenting flash-message copy)

## Audit asks for GPT

1. **Helper placement** ‚Äî is `src/lib/memory/admin-auth.ts` the right
   home for the decision, or should this live alongside `hasRole` in
   `src/lib/auth.ts`? My call: memory-specific decision belongs next to
   the memory modules, not in the generic auth surface; generalising to
   a reusable shape feels premature.
2. **Cross-team admin delete** ‚Äî I kept admins allowed cross-team to
   match P16-E's cross-team admin READ. Acceptable, or should the
   delete be tenant-gated even for admins (i.e. require membership
   even at admin role)?
3. **Flash-copy granularity** ‚Äî two distinct reject messages
   ("Editor role required to remove durable memory." vs "Not authorised
   for that team."). Clear enough as a governance hint, or is a single
   generic "Refused" cleaner?
4. **Render gate** ‚Äî the Remove button is hidden for viewers. Is
   hide-completely the right affordance, or should they see a disabled
   button with a tooltip explaining why? My call: hide keeps the row
   cleaner and the server gate is the real safety check.
5. **Empty-state copy** ‚Äî is *"will appear here once they're created"*
   tight enough, or would something more explicit like
   *"No operator tooling currently writes durable memory ‚Äî entries
   will appear here once a write path is shipped"* be clearer about
   the current state of the product?
