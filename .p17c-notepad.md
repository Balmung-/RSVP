## Pilot target

**Template:** `moather2026_moather2026`
**Language:** `ar`
**Shape:** `WhatsAppTemplateMessage` with
`headerDocument = { kind: "id", mediaId, filename }`

Meta template approval is already in place as of 2026-04-21.
Taqnyat token is wired via `TAQNYAT_WHATSAPP_TOKEN` (P17-A).

The operator's pilot flow is:
1. Upload the invitation PDF (reuses existing `/api/uploads`
   route — no new upload surface).
2. Configure the campaign to attach that PDF on WhatsApp sends
   (the field this slice adds to the `Campaign` model).
3. Ask `/chat` to send. `propose_send` shows doc readiness
   (template name + filename). `confirm_send` fires the upload-
   to-media step (P17-B) just-in-time, then sends via P17-A's
   `headerDocument` branch. Invitation.payload records the
   template name + filename + mediaId for audit.
4. Delivery outcome surfaces live in the same `/chat` session
   via the bridge GPT landed at `fcc8d63`.

## Scouting findings

### PDF byte source
`/api/uploads` already:
- accepts multipart FormData with `file` + optional `kind="doc"`
- persists bytes as `FileUpload.contents` (Postgres Bytes blob)
- returns `{ id, url: "/api/files/{id}", filename }`
- auto-extracts PDF text via the ingest pipeline
  (`src/lib/ingest/pdf.ts`)

No new upload surface needed for the pilot. The operator uses
the existing route; P17-C just connects a `FileUpload.id` to a
`Campaign`.

### Campaign-config surface
Currently: `brandLogoUrl`, `brandHeroUrl` (URL-only, not a
mediaId carrier), `CampaignAttachment[]` (URL-only sibling rows,
not suitable for a WhatsApp upload ref). **Net-new:** a nullable
FK from `Campaign` to `FileUpload`.

One field is enough for the pilot. Per-campaign template-name /
language overrides are deliberately deferred — `moather2026_moather2026`
in `ar` is THE pilot template, and prematurely multiplying
configurability for a single-template pilot violates directive A
("keep the pilot surface narrow").

### Invitation.payload provenance
`payload: String?` — JSON-encoded, written by `src/lib/delivery.ts`.
WhatsApp writers today encode `{ template, language, variables }`
for the template path; the `document` branch (`delivery.ts:288-304`)
is already reserved in the planner's output discipline but is
unreachable today.

No current reader of `payload` — it's a write-only audit surface
today. That means schema stability of the JSON shape is important
(future readers will depend on it) but no migration of downstream
consumers is needed when the document branch becomes reachable.

### Gate-3 acceptance criteria (from Agent chat.md, synced in
      `af7e6f2`)
- `sendCampaign`, resend, stages, and chat-confirmed sends must
  all converge on the SAME document send branch — no hidden
  one-off path.
- Invitations must persist template name + document ref in
  `Invitation.payload` for audit.
- Channel-scoped: the doc branch fires only when the campaign
  has WhatsApp-doc delivery configured.
- `propose_send` / `confirm_send` directives must show doc
  readiness + exact template identity; they must BLOCK if the
  PDF / template / doc ref is missing on a doc-configured
  campaign.

## Proposed sub-slice breakdown

Each sub-slice is independently committable + auditable. Earlier
sub-slices are pure / additive; the send-time plumbing lands only
after the upstream surfaces are in place.

- **P17-C.1 — schema + pure predicate.** Adds
  `Campaign.whatsappDocumentUpload` (nullable FK to `FileUpload`).
  Exports a pure `campaignWantsWhatsAppDocument(...)` predicate
  co-located with the existing WhatsApp planner. NO planner
  branch yet — the predicate is the eventual call site's gate,
  and pinning its semantics in isolation is worth its own slice.
  Strictly additive. Zero behavior change.

- **P17-C.2 — planner document branch.** Wires the planner's
  currently-unreachable `document` output. Campaign with the new
  FK + template name + language produces a `WhatsAppTemplateMessage`
  with `headerDocument = { kind: "link", link: "/api/files/{id}" }`
  (placeholder — the confirm-time upload in C.3 swaps it for a
  `{ kind: "id", mediaId }` ref). Pins the new planner branch
  against the predicate's outcomes.

- **P17-C.3 — just-in-time media upload on confirm.** The chat
  confirm_send handler intercepts plans with a `kind: "link"`
  headerDocument, reads the FileUpload bytes, calls
  `taqnyatUploadMedia(...)` (P17-B), and swaps the ref to
  `{ kind: "id", mediaId, filename }`. On upload failure, the
  send is refused with a structured error the chat widget can
  render. Pins the transition + the refuse-on-upload-fail path.

- **P17-C.4 — send-path persistence of doc provenance.** The
  delivery writer extends the WhatsApp `payload` JSON to include
  `documentMediaId` + `documentFilename` + `documentUploadId` in
  the template branch (the doc-enriched variant). Pins the
  provenance shape.

- **P17-C.5 — propose_send / confirm_send widget readiness.**
  The existing widgets get a doc-readiness line ("Will attach
  PDF: <filename>") and block when the campaign is doc-
  configured but the FileUpload is missing / zero-bytes / the
  template name is missing. Pins the blocker text + the
  no-silent-send discipline.

- **P17-C.6 (optional, only if scouting in C.5 reveals it) —
  send-blockers integration.** If the existing send-blockers
  module (`src/lib/ai/tools/send-blockers.ts`) needs updating
  so blockers aggregate correctly, that lands here. Otherwise
  merged into C.5.

### What stays outside P17-C

- **No admin surface widening.** The existing campaign admin
  (if it surfaces the new field at all) is a separate ticket.
  Directive A makes chat the sole primary surface; admin only
  exists for one-time config that an operator does out-of-band.
- **No chat-scoped file upload trigger.** Operators upload PDFs
  via the existing `/api/uploads` route. Adding a "upload this
  PDF" chat tool is a directive-A-respecting future slice but
  not needed for the first pilot proof.
- **No webhook-driven live-delivery-state push.** The `fcc8d63`
  bridge covers operator-triggered confirm/import live updates.
  Delivered/bounced pushed-via-webhook into an open chat session
  is still a separate seam and not a P17-C blocker (the operator
  can see delivery outcomes in the confirm_send result card that
  DOES live-update today).
- **No per-campaign template-name override.** Single pilot
  template only. Per-campaign overrides land if a second client
  needs a different template.
- **No persistence of the mediaId.** Meta retains uploaded
  media for 30 days. C.3 re-uploads on each confirm_send —
  simple, correct, and well within the retention window for the
  invitation flow. If a future flow needs longer retention (e.g.
  batched sends over weeks), that's its own scope.

## P17-C.1 scope — this commit

### Schema
Add to `Campaign`:

```prisma
// WhatsApp document-header delivery (P17-C). When set, the send
// planner uses the template's header-document component to attach
// this FileUpload as a PDF. Null = plain template (P17-A path).
whatsappDocumentUploadId String?
whatsappDocumentUpload   FileUpload? @relation("CampaignWhatsAppDocument", fields: [whatsappDocumentUploadId], references: [id], onDelete: SetNull)
```

Back-relation on `FileUpload`:

```prisma
whatsappHeaderCampaigns Campaign[] @relation("CampaignWhatsAppDocument")
```

`onDelete: SetNull` is the right discipline: if the operator
deletes the FileUpload, the campaign falls back to the plain
template path rather than cascading into destruction. The plain
path is the safe-but-degraded fallback.

### Pure predicate
`src/lib/providers/whatsapp/sendPlan.ts` exports:

```ts
export type WhatsAppDocumentGateInput = {
  whatsappDocumentUploadId: string | null;
  templateWhatsAppName: string | null;
  templateWhatsAppLanguage: string | null;
};

export function campaignWantsWhatsAppDocument(
  input: WhatsAppDocumentGateInput,
): boolean;
```

Returns `true` iff ALL THREE fields are non-null and non-empty.
A campaign with a document upload id but no template name is
configuration-broken: the doc header can only attach to a
template, so the predicate refuses the combination.

Co-located with the planner because P17-C.2 will call it as the
planner's first branching check. Keeping the predicate separate
(vs. inlining into `decideWhatsAppMessage`) means P17-C.2's
planner change is small and the predicate's semantics are
independently pinnable.

### Pins (5 new)

In a new `tests/unit/whatsapp-document-gate.test.ts`:

1. All three fields present + non-empty → `true`.
2. Missing `whatsappDocumentUploadId` (null) → `false` (nothing
   to attach).
3. Missing `templateWhatsAppName` (null) → `false` (doc header
   requires a template).
4. Missing `templateWhatsAppLanguage` (null) → `false` (Meta
   keys templates on name + language pair).
5. Empty-string field (length-0, but non-null) → `false` —
   mirrors the planner's existing length-0 discipline at
   `sendPlan.ts:93-96`.

### Non-changes in this slice

- NO planner branch change. `decideWhatsAppMessage` still
  returns `kind: "template"` without a headerDocument. P17-C.2
  wires the branch.
- NO call-site change. `WhatsAppPlanCampaign` shape is
  unchanged — the predicate takes its own narrow input shape.
  (P17-C.2 will widen `WhatsAppPlanCampaign` to include the new
  field.)
- NO chat or widget change.
- NO delivery or payload change.
- NO admin UI change.
- NO migration file created (repo uses `prisma db push`; the
  schema.prisma edit IS the ship artifact).

### Audit asks for GPT on this sub-slice

1. **Predicate input shape.** I used a narrow local shape
   (`WhatsAppDocumentGateInput`) rather than taking a full
   `WhatsAppPlanCampaign`. Rationale: the predicate has a single
   responsibility (the three-field gate) and a narrow input
   makes it reusable outside the planner context. Agree, or
   would you rather it take the same `WhatsAppPlanCampaign` the
   planner uses, so P17-C.2's callers don't double-project?

2. **`onDelete: SetNull` vs. the alternatives.** I went with
   SetNull because the plain template path is a safe fallback.
   Alternatives: `Cascade` (delete the campaign) — too aggressive;
   `Restrict` (block the FileUpload delete) — operator-hostile.
   Agree?

3. **Single nullable FK vs. a sibling config row.** I'm storing
   the doc ref as a single nullable column on `Campaign`. The
   alternative was a `CampaignWhatsAppConfig` sibling row that
   could later carry per-campaign template overrides, caching,
   etc. My read: premature — one column is enough for the
   single-template pilot, and widening to a sibling row later
   is a mechanical refactor. Agree, or prefer the sibling now?

4. **Predicate scope.** The predicate ignores whether the
   FileUpload actually exists or has non-zero bytes. Those
   checks happen at confirm-time in C.3 (the upload step will
   catch empty bytes via the short-circuit guard pinned in
   P17-B). The gate here is purely "is the campaign CONFIGURED
   for doc delivery," not "will the send work." Agree with that
   split, or should the gate also walk the FK?

5. **Sub-slice size.** This slice is deliberately tiny (one
   schema field + one pure predicate + five pins). Is this the
   right atomic size for the P17-C cadence, or would you prefer
   a fatter first slice that also lands the planner branch
   (C.1 + C.2 together)?
