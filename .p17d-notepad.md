# P17-D notepad — campaign-edit UI for WhatsApp template + PDF

Follows-from: `ca472d1` (GPT's P17-C closeout verdict +
P17-D direction synced).

Follows-tranche: `29a175b` (P17-C tranche shipped — runtime
WhatsApp-PDF send path works end-to-end, but operators still
can't configure it from the product surface).

## Tranche purpose

Close the last remaining gap in the WhatsApp-PDF pilot: give
operators a real form on the normal campaign UI to configure the
four fields the runtime now depends on. Before P17-D, a campaign
could only be wired for the WhatsApp template + PDF via Prisma
Studio or a migration — the chat-path propose/confirm/send flow
shipped in P17-C is functionally unreachable.

Explicit chat-first justification (per GPT's P17-D direction sync
in Agent chat.md ≈L15413–L15423): this tranche is allowed only
because it unlocks `/chat`'s already-shipped WhatsApp-PDF send
path. This is NOT a broad admin/manual-send widening.

## Scouting findings

### Affected fields

All four are already on the `Campaign` table:

- `templateWhatsAppName` — Meta template identifier, exact-match
  (e.g. `moather2026_moather2026`).
- `templateWhatsAppLanguage` — BCP-47 tag (e.g. `ar`, `en_US`).
- `templateWhatsAppVariables` — JSON-encoded positional expression
  array (e.g. `["{{name}}", "{{venue}}"]`) or null for zero-var
  templates.
- `whatsappDocumentUploadId` — nullable FK to `FileUpload`, added
  in P17-C.1.

### What exists today

- **`src/components/CampaignForm.tsx`** — shared form used by
  `/campaigns/new` and `/campaigns/[id]/edit`. Has disclosure
  sections for Branding + Templates (email subject/body + SMS
  body). No WhatsApp fields.
- **`src/app/campaigns/new/page.tsx`** — inline `createCampaign`
  server action. Reads FormData fields, writes via
  `prisma.campaign.create`. No WhatsApp handling.
- **`src/app/campaigns/[id]/edit/page.tsx`** — inline
  `updateCampaign` server action. Mirror of create's reads + an
  `update`. No WhatsApp handling.
- **`src/components/FileInput.tsx`** — drag-drop file picker that
  posts to `/api/uploads` and writes the result **URL** into a
  hidden form field. Used today for `brandLogoUrl` / `brandHeroUrl`
  (URL-column fields).
- **`/api/uploads`** — already accepts `kind="doc"` (which permits
  PDF via `DOC_MIMES`). Returns `{ id, url, filename, ingest }`.
  Role-gated to editor.
- **RBAC discipline** — both server actions `requireRole("editor")`
  and gate the target campaign via `canSeeCampaign(...)`.
  Extending the write surface inherits both gates automatically.

### What's missing

- Parser + write for the 4 WhatsApp fields on both create + edit
  server actions.
- Form inputs for the 3 text fields (name / language / variables).
- A file picker that stores the FileUpload **id** (not the URL)
  for `whatsappDocumentUploadId`. `FileInput` can't do this
  without a second mode — existing callers depend on URL
  semantics.

### Gap check — does the chat-path auto-pick up?

Yes, by construction:
- `propose_send` + `send_campaign` already `select` all four
  fields (P17-C.5).
- `computeBlockers` already gates on them (P17-C.5).
- `performWhatsAppSend` already resolves them at delivery edge
  (P17-C.2/3/4).
- Once the form writes the values to the Campaign row, the
  existing chat-path reads them on the next `propose_send` call.

No `/chat`-side changes needed in this tranche — the live-update
bridge (`fcc8d63`) makes the new values visible in-session as
soon as they're saved.

## Strict scope (GPT — Agent chat.md L15418–L15423)

1. Campaign create/edit surface for WhatsApp template config.
2. Campaign create/edit surface for PDF/document assignment.
3. Server action / parse / RBAC wiring for those fields.
4. Chat-first: allowed only because it unlocks the shipped
   `/chat` WhatsApp-PDF send flow.
5. **Do NOT** widen unrelated admin / manual-send surfaces. The
   legacy `/campaigns/[id]/page.tsx` (detail view), the manual
   send buttons, and the workspace dashboard WhatsApp rollup all
   stay untouched by this tranche unless a hard correctness
   dependency forces an edit — and even then, minimal.

## Acceptance bar (GPT L15426–L15428)

- An operator can configure the approved WhatsApp template + PDF
  on a campaign from the normal campaign UI (`/campaigns/new` +
  `/campaigns/[id]/edit`).
- `/chat` can then propose, confirm, and send that campaign
  without a page refresh, with the same open session updating in
  place.
- Missing / removed PDF state fails closed and is visible before
  send (already covered by the `no_whatsapp_document` blocker
  shipped in P17-C.5).

## Proposed sub-slice breakdown

Slicing mirrors P17-C's rhythm: each slice is independently
committable + auditable; earlier slices are code-only foundations
that later UI slices sit on top of.

- **P17-D.1 — server action + pure parser.** Adds pure helper
  `parseWhatsAppCampaignFields(fd)` in
  `src/lib/campaigns/whatsapp-form.ts` that reads the 4 fields
  from FormData with trim + length-clip + null-if-empty. Wires
  the helper into `createCampaign` + `updateCampaign` server
  actions. FK existence check for `whatsappDocumentUploadId`
  runs inline in the server action (Prisma lookup; dangling id →
  drop write, matching the existing `if (!name) return;`
  convention in createCampaign). No UI change — the form still
  has no fields; the server side just learns to handle them when
  they arrive.

- **P17-D.2 — UI text fields for template config.** Adds
  templateWhatsAppName / templateWhatsAppLanguage /
  templateWhatsAppVariables fields to `CampaignForm.tsx` under
  the existing "Templates — override defaults" disclosure (or a
  parallel "WhatsApp template" disclosure if it gets too
  crowded). No file picker yet. Operator can now configure a
  WhatsApp template but can't assign a PDF.

- **P17-D.3 — PDF picker UI.** Adds a new
  `WhatsAppDocumentInput` client component (sibling to
  `FileInput` but writes the FileUpload **id**, not URL; doc-only
  accept filter; no URL-paste fallback because Meta needs our
  upload). Wires into CampaignForm next to the template fields.
  This closes the acceptance bar.

If any slice surfaces unexpected scope, it grows its own sub-
slices (e.g. D.3.1, D.3.2) — the P17-C pattern.

## Design calls locked into D.1

- **Parser is pure + sync.** Shape-only validation (trim / clip /
  null-if-empty). No JSON parse of `templateWhatsAppVariables` at
  write time — the send-blocker's `template_vars_malformed`
  already catches malformed JSON at send time, with the raw
  string stored so the operator can fix it in the edit form. This
  preserves the existing pattern where `templateEmail` stores
  arbitrary text (no validation) and errors surface at preview /
  send time.

- **FK existence check at server-action level, not parser.** The
  parser stays pure + DB-free so it unit-tests trivially.
  `createCampaign` / `updateCampaign` do the
  `prisma.fileUpload.findUnique({ where: { id } })` themselves —
  mirrors how propose_send / send_campaign already do the
  existence check in their handlers (not inside
  `computeBlockers`).

- **Dangling FK = drop the write silently.** If the operator
  submits a non-existent FileUpload id, the server action writes
  `null` for that field. Reasoning:
  - Consistent with the existing `if (!name) return;` silent-drop
    convention on bad input in `createCampaign`.
  - The `no_whatsapp_document` blocker (P17-C.5) already surfaces
    the drift visibly on the next `propose_send` — if the
    operator also set template name + language, they'll see
    "WhatsApp invitation PDF is missing" on the ConfirmSend card.
  - A dangling FK write would violate Prisma's FK constraint at
    INSERT time anyway — silently nulling is the clean
    alternative to a 500.

- **Length caps on text fields match the schema / existing
  patterns.**
  - name: 200 chars (matches `name String` default discipline +
    Meta's name limit).
  - language: 10 chars (BCP-47 worst-case `zh_Hant_HK` = 10).
  - variables: 2000 chars (generous for a JSON array of ~20
    expressions, well under Meta's BODY param total cap).
  - uploadId: 50 chars (cuid is 25 chars; 50 is safe with
    headroom).

- **Permissive write: both-or-neither is NOT enforced at write
  time.** A campaign can save with name-only or language-only.
  The `no_whatsapp_template` blocker surfaces the gap at send
  time. Mirrors how `templateEmail` can save without
  `subjectEmail`.

## What doesn't land in D.1

- No UI changes (D.2 + D.3 cover those).
- No flash-message system for write-time errors. Dangling FK +
  other bad input use the silent-drop convention.
- No JSON validation of variables at write time (delegated to
  send-blocker's existing check).
- No per-team scope on the FileUpload id. Editors can reference
  any existing upload in the system (mirrors current FileUpload
  access — no per-team upload scope exists anywhere in the
  codebase as of 2026-04-21).
