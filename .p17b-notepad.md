## What this slice does

### Two new exports from `src/lib/providers/whatsapp/taqnyat.ts`

```ts
export async function taqnyatUploadMedia(
  opts: TaqnyatMediaUploadOptions,
): Promise<TaqnyatMediaUploadResult>;

export function buildMediaUploadFormData(
  bytes: Uint8Array,
  filename: string,
  mimeType: string,
): FormData;
```

Plus the types `TaqnyatMediaUploadOptions` and
`TaqnyatMediaUploadResult` (symmetric with `SendResult`).

The uploader hits `POST https://api.taqnyat.sa/wa/v2/media/`
with multipart body + `Authorization: Bearer <token>`. On a
2xx with extractable id it returns
`{ ok: true, ref: { kind: "id", mediaId, filename } }`;
otherwise `{ ok: false, error, retryable? }` with 5xx → retryable
and 4xx → non-retryable.

### Five design calls worth naming

1. **Standalone export, NOT a method on `WhatsAppProvider`.**
   Media upload is Taqnyat-specific. Putting it on the interface
   would either pollute every stub with throw-or-fake branches
   or force callers through an optional-method check. P17-C can
   branch on `getWhatsAppProvider().name === "taqnyat-whatsapp"`
   when it needs to decide whether upload is available.

2. **Exported `buildMediaUploadFormData` is a pure helper,
   testable WITHOUT fetch.** The existing test stub in
   `taqnyat-whatsapp.test.ts` treats bodies as strings
   (`typeof init?.body === "string" ? init.body : undefined`) —
   it can't introspect a FormData body. Rather than widen the
   shared stub, the P17-B tests have their own FormData-aware
   fake in `taqnyat-media-upload.test.ts` AND directly pin the
   FormData field set via the pure helper. Both paths tested.

3. **Content-Type is NEVER set manually.** The FormData
   serializer writes `multipart/form-data; boundary=<random>`
   when the body passes through fetch; setting a static
   `Content-Type: multipart/form-data` drops the boundary and
   Meta's multipart parser rejects the request. A pinned test
   catches a future refactor that adds the header.

4. **Response tolerance: flat `{ id }` primary + `{ media: { id } }`
   fallback.** Taqnyat proxies Meta's flat shape today. If they
   ever wrap it (as some BSPs do), the uploader still works.
   We don't depend on the wrapper being present.

5. **Short-circuit guards before fetch.** Missing token /
   filename / mimeType / empty bytes each returns an error
   WITHOUT issuing a fetch call. Filename is especially load-
   bearing: without it the returned ref is unusable at send
   time because both `WhatsAppDocumentMessage` and
   `WhatsAppTemplateMessage.headerDocument` carry filename
   forward to the recipient-visible envelope.

### Media-expiry note

Meta retains uploaded media ids for 30 days. For the invitation-
PDF flow (which sends the document shortly after upload) this
is comfortable headroom. If a future flow needs longer retention,
P17-C (or its successor) will re-upload on a schedule. We do
NOT cache / persist ids here — that's a P17-C decision.

## What's pinned by tests (21 new pins)

All in `tests/unit/taqnyat-media-upload.test.ts`.

**FormData shape (6):**
1. `messaging_product='whatsapp'` — Meta requires this literal.
2. `type=<mimeType>` for PDF.
3. `type=<mimeType>` for PNG — proves mimeType isn't hardcoded.
4. File part has filename + MIME type on the multipart part
   (pinned so Meta doesn't derive filename from nothing).
5. File part preserves exact byte content — anti-drift pin
   against a string-coercion refactor corrupting binary.
6. Exactly three fields — no silent widening (e.g. a phone-
   number-id field copied from the send envelope) and no silent
   shrinking (file part must remain).

**Uploader transport (6):**
7. Happy path returns `{ ok: true, ref: { kind: "id", mediaId, filename } }`
   as a deepEqual — ref shape is the exact input to send-time
   envelopes, so drift here trips loudly.
8. URL is `https://api.taqnyat.sa/wa/v2/media/` (trailing slash).
9. POST method.
10. `Authorization: Bearer <token>`.
11. Content-Type is NOT manually set (case-insensitive check —
    some fetch implementations normalise header casing).
12. Body is a `FormData` instance (type-level pin).

**Response-shape tolerance (4):**
13. Tolerates `{ media: { id } }` nested shape.
14. 200 with NO id in any shape → `ok:false` (refuse-to-
    fabricate).
15. 500 → `retryable:true` + error text carried through.
16. 4xx → `retryable:false` + error text carried through.
17. Malformed JSON body → `ok:false` (parse error swallowed;
    falls into no-id path).

**Short-circuit guards (5):**
18. Missing token → `ok:false`, NO fetch call.
19. Missing filename → `ok:false`, NO fetch call (error
    mentions `filename`).
20. Missing mimeType → `ok:false`, NO fetch call (error
    mentions `mimeType`).
21. Empty bytes → `ok:false`, NO fetch call (error mentions
    `empty`).
22. Uses `globalThis.fetch` when `fetchImpl` is absent —
    exercises the default-parameter branch.

(Yes, I numbered 22 items but said 21; 22 was the global-fetch
fallback case — the suite reports "1611 / 1611" which is the
prior 1590 plus the new 21 discrete `test(...)` blocks. The
global-fallback case is a single test.)

## Non-changes in this slice

- **No send-path wiring.** `taqnyatWhatsApp.send` is unchanged;
  no code path calls `taqnyatUploadMedia` yet. P17-C will wire
  it in.
- **No chat-path or UI change.** Zero new routes, zero new
  server actions, zero page/widget changes.
- **No schema / migration / RLS change.**
- **No new env vars.** `TAQNYAT_WHATSAPP_TOKEN` is resolved by
  the provider factory and passed explicitly into
  `taqnyatUploadMedia` by callers. P17-C will decide whether
  this needs a helper or if the factory should export a
  token-bound upload method.
- **No `uploadMedia` on the `WhatsAppProvider` interface.**
  Deferred until multiple providers need it; premature shape
  promotion is worse than a second, late refactor.
- **No touch to stubWhatsApp.** The stub provider remains a
  `send()`-only shim.

## Audit asks for GPT

Five for this slice.

1. **Standalone export vs interface method.** I kept
   `taqnyatUploadMedia` off `WhatsAppProvider` on the argument
   that media upload is Taqnyat-specific. The downside: P17-C
   will need to import directly from the Taqnyat module (or
   type-guard via provider.name). Agree with the call, or
   would you prefer an optional `uploadMedia?` on the interface
   with a stub no-op, for cleaner provider-factory wiring?

2. **Pure-helper export + FormData-aware fake.** I exported
   `buildMediaUploadFormData` specifically so the field-set
   pins don't need a FormData-aware HTTP stub. The uploader
   tests have their own FormData-widened fake. Does the
   helper-export boundary read clean to you, or would you
   prefer the helper stay internal and the HTTP stub do all
   the multipart parsing?

3. **Short-circuit guards fail-closed WITHOUT throwing.**
   Missing token/filename/etc. return `{ ok: false, error }`
   rather than throwing. Consistent with the send path but
   arguably a missing-token is a config bug (should throw) vs
   an empty-bytes is a caller bug (should return an error).
   Agree with the uniform-shape choice, or should config bugs
   throw so they surface in logs as stack traces?

4. **Filename validation is length-0 only.** No MIME-to-
   filename consistency check (e.g. rejecting `invite.pdf` +
   `mimeType: image/png`). My read: Meta accepts whatever, and
   forcing consistency here would be premature validation.
   Agree, or should the helper reject obvious mismatches?

5. **`fetchImpl` DI seam.** I used a parameter for test
   injection rather than a module-level mutable (`_fetch`) or
   a factory override. The parameter makes each call site
   explicit about who owns the fetch. Agree, or prefer a
   module-level seam so real callers don't thread anything?

## Next step

**P17-C — chat-path wiring for WhatsApp PDF sends.**

Per GPT's `f698229` greenlight + `af7e6f2` directives:

- Target surface is `/chat` only (no admin/manual-send widening).
- Chat flow: operator asks via chat → workspace shows readiness
  → confirm/send happens from the workspace action → delivery

  `Agent chat.md`).
