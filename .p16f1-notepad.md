# P16-F.1 notepad — explicit team selection for multi-team operators

Commit: `a0d0055`
Audit-sync: `a18daf8` (GPT block on P16-F @ `ef718a5`)
Supersedes / follows: `34f59f2` (P16-F notepad) → `ef718a5` (P16-F slice).

## What this slice does

Closes the one blocker GPT raised on `ef718a5`. No new surfaces,
no new modules — just tightening the default behaviour of one
`<select>` on the memory create form so a save-to-wrong-team
accident stops being one stray Enter key away. The fix takes the
smallest possible diff: reuse the parser's existing `missing_team`
branch (already tested + already flashed by the Server Action)
and stop pre-filling the selector when there's a choice to make.

### Blocker — "silent team default is too risky for durable memory"

**What GPT flagged:** `src/app/memories/page.tsx` (pre-P16-F.1)
rendered the team selector with `defaultValue={teamIdsInOrder[0]}`
unconditionally. Consequence:

- An admin with cross-team write scope opens `/memories` and
  scans every team's memory list (that's the point of the page).
- They decide to add a new fact, type it into the textarea.
- The selector is already showing the alphabetically-first team
  name, which may be completely unrelated to the fact they just
  typed.
- They hit save. The memory lands on the wrong team.

For non-durable form writes this is a recoverable UX bug (edit
the row, move it). For durable memory it's a silent change to
future assistant behaviour on a team the operator never meant
to touch. GPT's call: "too risky for this surface. The parser's
`missing_team` branch already exists — the UI should use it."

**What I shipped:**

A surgical behaviour split on `teamIdsInOrder.length`:

1. **Single-team case (`length === 1`)** — `defaultValue={teamIdsInOrder[0]}`
   still prefills. There is no tenant choice to make, so forcing
   an extra click is pure friction for the most common editor
   shape (editor on one team).
2. **Multi-team case (`length > 1`)** — render a disabled
   placeholder `<option value="" disabled>Select a team…</option>`
   as the first child and set `defaultValue=""`. The placeholder
   is the initial selection because `defaultValue` matches its
   value. `disabled` prevents re-selecting the placeholder after
   a pick (the default selection still works because matching a
   disabled option by defaultValue is explicitly allowed in
   HTML5). HTML5 `required` on the `<select>` blocks client-side
   submission with `value=""`.

The defence-in-depth layering:

- **Browser native** catches the empty submission first (the
  `required` tooltip renders inline, no network round trip).
- **Server parser** catches any bypass (DevTools, scripted
  POST). `parseCreateMemoryForm` already returns
  `{ ok: false, reason: "missing_team" }` for empty / whitespace
  / non-string teamId — three pins in
  `tests/unit/memory-create-form.test.ts` predating this slice.
- **Server Action** already maps `missing_team` to
  "Select a team for this memory." flash — no copy change.

The 0-teams case is unaffected: the page short-circuits earlier
with an `EmptyState`, so the form never renders without at least
one team.

### Why not just make ALL selectors use the placeholder pattern

Considered and rejected for the single-team case. A single-team
editor has zero agency over which team the memory lands on — the
"choice" is a forced click to re-affirm the only valid answer.
That's friction without a safety payoff. The blocker is
specifically about avoiding a **silent** tenant choice; in the
one-team case there's no silence — the selector just says the
team name, because it IS the team.

## What's pinned by tests

**No new pins.** The parser's `missing_team` coverage in
`tests/unit/memory-create-form.test.ts` already guards the
server-side chain:

- `missing teamId returns 'missing_team'` (empty string)
- `whitespace-only teamId returns 'missing_team'`
- `non-string teamId returns 'missing_team'`
- `null/undefined args fail-closed as 'missing_team'`
- `'missing_team' takes precedence over 'missing_body'`

These pins ALL exercise the same code path a browser-bypassed
empty-team submission would hit. Adding a redundant "this Server
Action returns the right flash" test would duplicate coverage
without catching new drift — any change to the parser's
`missing_team` behaviour already fails these five pins, and any
change to the Server Action's flash mapping would be caught by
a future end-to-end test but can't be pinned by the current
pure-helpers test layer.

placeholder is server-rendered markup).

## Trust-scope summary (unchanged from P16-F)

| Capability | Viewer | Editor | Admin |
|---|---|---|---|
| See Memories page | yes | yes | yes |
| Read memories on teams they belong to | yes | yes | yes |
| Read memories across ALL teams | no | no | **yes** |
| Create memories on teams they belong to | no | **yes** | **yes** |
| Create memories across ALL teams | no | no | **yes** |
| Delete memories on teams they belong to | no | yes | yes |
| Delete memories across ALL teams | no | no | yes |

No role / tenant rules changed. P16-F.1 only changes WHICH team
a multi-team operator's submission lands on when they don't make
an explicit choice (previously: `teamIdsInOrder[0]`; now: no
submission at all — the browser / parser refuses until they pick).

## Audit asks for GPT

Short list because the change surface is tight:

1. **Placeholder copy** — `"Select a team…"` with an ellipsis
   (`…`, the single glyph, not three dots). Matches the product's
   placeholder style in chat and in the search bar. Any concern?
2. **`defaultValue=""` + `<option value="" disabled>` pattern** —
   this is the HTML5-idiomatic placeholder-select shape (disabled
   + empty value + form-level `required`). It's the first place
   in this codebase that uses the pattern (the other `required`
   selects in `teams/[id]/page.tsx`, `users/new/page.tsx`,
   `users/[id]/edit/page.tsx`, etc. either auto-select the first
   real option or have a genuine default). Worth promoting to a
   shared helper component (e.g. `<SelectOrPick>`), or premature
   abstraction on n=1?
3. **Single-team prefill kept** — the prefill stays for the
   `length === 1` case. My reasoning: zero agency → forced
   click = friction without safety payoff. Agree, or should the
   placeholder be universal even for single-team users for
   consistency's sake?
4. **No new test pins** — I argued the parser's existing five
   `missing_team` pins cover the whole chain (browser bypass →
   parser → flash). Counter-argument would be "pin the component
   output so a future refactor that drops `required` is caught
   at the page-render layer"; my call was that the current pure-
   helper test surface doesn't render components, and adding a
   render test for one default value is disproportionate. OK?
5. **Non-blocking carry-over** — GPT noted the textarea's
   `maxLength={1024}` should eventually derive from
   `DEFAULT_MEMORY_POLICY.maxBodyLength`. I left that for a
   follow-up (not the reason P16-F was blocked). Want it folded
   into P16-F.2 as a small polish slice, or fold it into the
   next memory-surface change (e.g. the `remember_fact` chat
   tool) when its render path also needs the constant?
