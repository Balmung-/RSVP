## Tranche goal (P17)

The WhatsApp document/PDF send path is the last materially-missing
product capability on the roadmap (line 5714 in `Agent chat.md`).
The use case that drives it is sending an approved invitation PDF
to a recipient over WhatsApp as a business-initiated conversation,
i.e. a pre-approved template with a DOCUMENT header carrying the
PDF.

The tranche is planned as three slices so each stays reviewable in
isolation:

- **P17-A (this slice)** — types + adapter request shape.
  Type-level split `WhatsAppDocumentMessage` vs
  `WhatsAppTemplateMessage.headerDocument`, plus the Taqnyat
  adapter body shaping for both. **No media upload, no send-path
  wiring.**
- **P17-B** — the `/wa/v2/media/` upload seam. Separate adapter
  method that returns a `WhatsAppDocumentRef { kind: "id", mediaId }`
  from a local byte buffer. Keeps P17-A independently testable
  with link-or-id references that callers can synthesise.
- **P17-C** — invitation flow wiring: campaign config surface for
  a PDF attachment, upstream-of-planner routing to the document
  message shape, rendering + provenance.

This slice (P17-A) only lands the foundation — types and request
shaping. The other two slices will land when the accountant's
template is Meta-approved (still pending as of today).

## What this slice does

### Type layer (`src/lib/providers/types.ts`)

Three additions, one extension, one union widen:

```ts
export type WhatsAppDocumentRef = { filename?: string } & (
  | { kind: "id";   mediaId: string }
  | { kind: "link"; link: string }
);

export interface WhatsAppDocumentMessage {
  kind: "document";
  to: string;
  document: WhatsAppDocumentRef;
  caption?: string;           // standalone-only
}

export interface WhatsAppTemplateMessage {
  // ...existing fields...
  headerDocument?: WhatsAppDocumentRef;   // NEW
}

export type WhatsAppMessage =
  | WhatsAppTextMessage
  | WhatsAppTemplateMessage
  | WhatsAppDocumentMessage;              // widened
```

Two design points worth naming:

- **`WhatsAppDocumentRef` is discriminated on `kind`, not on which
  field is present.** An `{ id?, link? }` shape with a runtime
  check is easier to construct but lets a caller pass both at
  once — the adapter would then silently pick one. A discriminated
  `kind: "id" | "link"` makes the constructor choose, and lets
  the adapter branch once without re-checking.
- **`caption` lives on `WhatsAppDocumentMessage`, not on
  `WhatsAppDocumentRef`.** Meta's envelope rejects `caption` inside
  a header-component document parameter (caption is a standalone-
  message-only field). Putting caption on the ref type would let
  a caller set `template.headerDocument.caption` and have the
  adapter silently drop it at build time. Putting caption on the
  standalone-message type makes the Meta rule a type-level
  constraint: you cannot reach header-document path + caption from
  the same call site.

### Adapter layer (`src/lib/providers/whatsapp/taqnyat.ts`)

Three changes:

1. **`buildRequestBody` dispatches a new `document` branch.**
   ```ts
   if (msg.kind === "document") return documentBody(to, msg);
   ```
   The existing text / template branches are unchanged.

2. **New `documentBody(to, msg)`** emits Meta's standalone-
   document envelope:
   ```ts
   {
     messaging_product: "whatsapp",
     recipient_type: "individual",
     to,
     type: "document",
     document: documentRefEnvelope(msg.document, msg.caption),
   }
   ```
   Caption is passed to the envelope helper because the standalone
   shape DOES accept it.

3. **`templateBody` refactored to build components in Meta's
   expected order.** Previously it pushed a single `body` component
   when variables were present. Now it accumulates HEADER → BODY:
   ```ts
   const components: Array<Record<string, unknown>> = [];
   if (msg.headerDocument) components.push({ type: "header", ... });
   if (msg.variables?.length) components.push({ type: "body", ... });
   if (components.length > 0) template.components = components;
   ```
   Buttons (P11 deferred) would slot after BODY when added.

4. **Shared `documentRefEnvelope(ref, caption?)`** keeps the
   `id` / `link` / `filename` discipline identical for the
   standalone-message path and the template-header-document path —
   the only difference is whether caption is included (passed only
   on the standalone-message call).

### Exhaustive narrowing update (`src/lib/delivery.ts`)

The audit-payload formatter `payloadForPlan` was narrowing the
planner's message with `if (kind === "text") ... else template`.
Widening `WhatsAppMessage` to include document would break
TypeScript narrowing on the `else` branch, so the helper now
exhaustively handles the document arm:

```ts
if (plan.message.kind === "text") return plan.message.text;
if (plan.message.kind === "document") {
  return JSON.stringify({
    document: plan.message.document,
    caption: plan.message.caption ?? null,
  });
}
return JSON.stringify({ template: ..., language: ..., variables: ... });
```

Semantic note: `decideWhatsAppMessage` cannot produce a document
plan today (the planner reads campaign config fields that only
trigger text/template). The exhaustive narrowing is a forward-
compatibility measure: the invitation-PDF flow is expected to
wire *upstream of the planner* (P17-C), but if a later refactor
routes document through the planner, the audit surface stays
honest without further edits.

## What's pinned by tests (11 new + 1 extended)

All new pins in `tests/unit/taqnyat-whatsapp.test.ts`. Pure
request-shape coverage — the build function is pure so direct
coverage catches silent drift against Meta's JSON before a live
send does.

**Standalone-document pins (5):**
1. `standalone document (id) produces Meta document-by-id shape` — envelope has `type: "document"` and `document: { id: "wa-media-abc" }`.
2. `standalone document (link) produces Meta document-by-link shape` — envelope has `document: { link: "https://..." }`.
3. `standalone document without filename omits the field` — filename is optional; Meta derives from URL path.
4. `standalone document with empty-string filename omits the field` — treats empty-string same as absent, keeps envelope tidy.
5. `standalone document caption included only when non-empty` — both branches tested (caption present + caption empty/undefined omitted).

**Template-header-document pins (5):**
6. `template with headerDocument (id) emits header component` — components array has `[{ type: "header", parameters: [{ type: "document", document: { id } }] }]`.
7. `template with headerDocument (link) emits header component` — same shape but with `link`.
8. `template with headerDocument + variables → HEADER then BODY order` — explicit index check `components[0].type === "header"` + `components[1].type === "body"`. This pin catches the most common refactor drift (Meta tolerates neither arbitrary ordering nor missing components once a template is approved with a specific structure).
9. `template with headerDocument and NO variables omits body component` — `components.length === 1`, type is `header`.
10. `template headerDocument parameter does NOT carry caption` — defensive pin; even if a future refactor adds caption to `WhatsAppDocumentRef`, this test refuses to let the header path emit it.

**Regression pin (1):**
11. `template WITHOUT headerDocument or variables omits components entirely` — the build function should not emit an empty `components: []` key; Meta accepts template sends without the key for zero-parameter templates.

**Extended pin:**
- `WhatsAppMessage discriminates by kind at the type level` now includes the document arm — the three-way narrowing in the test body is itself the type-level contract.

## Non-changes in this slice

- **No media upload.** `/wa/v2/media/` is untouched; that's P17-B.
  Callers wanting to send by `mediaId` must synthesise the id
  externally until P17-B lands.
- **No send-path wiring.** `decideWhatsAppMessage` still emits
  only text / template. No campaign-config surface for PDF
  attachment yet — that's P17-C.
- **No stub changes.** `stubWhatsApp` already `send`s any
  `WhatsAppMessage` variant and returns synthetic success; the
  union widening is transparent to it.
- **No provider factory changes.** `getWhatsAppProvider()` routes
  on `WHATSAPP_PROVIDER` env as before.
- **No database schema change.** Document sends when wired will
  write to the same `Invitation` rows with `channel: "whatsapp"`
  and payload column carrying the new document descriptor JSON.
- **No policy/auth surface change.**

## Audit asks for GPT

Four for this slice.

1. **DocumentRef shape** — I put `kind: "id" | "link"` as the
   discriminator rather than `{ id?, link? }` with runtime check.
   My reasoning: forces the caller to commit to one path, lets
   the adapter branch once without re-checking. Agree, or does
   the optional-field shape read cleaner at call sites you'd
   expect (e.g. when a caller has ambiguous data and wants the
   adapter to choose)?
2. **Caption placement** — caption lives on
   `WhatsAppDocumentMessage`, not on `WhatsAppDocumentRef`. This
   makes the Meta "no caption on header-document" rule a type-
   level constraint. Agree that type-level enforcement is worth
   the minor shape asymmetry, or would you prefer a runtime drop
   with a comment?
3. **Component-order pin (#8)** — explicit index check
   `components[0].type === "header"` + `components[1].type === "body"`.
   I argued Meta tolerates neither arbitrary ordering nor missing
   components once a template is approved. Right level of
   strictness, or is the pin overfitted against Meta's actual
   behaviour (which I can't independently verify without a live
   template approval)?
4. **Exhaustive narrowing in payloadForPlan** — I made the
   helper handle the document arm even though the planner can't
   emit it today. The alternative was a type-level assertion
   `as { kind: "text" | "template" }` at the callsite, which is
   terser but hides the widening. Agree with the exhaustive
   read, or prefer the assertion-and-comment pattern?
