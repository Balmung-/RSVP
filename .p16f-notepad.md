## What this slice does

Closes the "no live callsite for the write seam" gap GPT called out as
residual when green-lighting P16-E.1: *"the actual memory-write path is
still deferred to the later P16 memory slices."* The P16-B write seam
(`createMemoryForTeam` in `src/lib/memory/server.ts`) has existed —
validated and unit-tested — since the first P16 slice, but had no live
callsite under `src/`. P16-F gives it its first sanctioned caller: an
editor-gated form on `/memories` that lets operators save durable facts
for the teams they govern. The `/memories` surface is now a complete
round-trip: operators can **read**, **create**, and **delete** durable
memory, all inside the one governed page.

The chat-tool write path (`remember_fact`) remains deferred for a
dedicated, injection-defended slice. See "Follow-ups" below.

## Design decisions

### Pure form parser — `src/lib/memory/form.ts`

The form helper converts a caller's `{ rawBody, rawTeamId, createdByUserId }`
(plus optional policy override) into either a canonical `MemoryWriteInput`
or a **tagged** reject reason. This is the one thin wrapper between
FormData and `createMemoryForTeam`. Two invariants:

- **Shape-compatible with the validator** — on `ok: true`, the returned
  `input` is guaranteed to pass `validateMemoryWrite`. The seam re-runs
  the validator on every write, so a regression here fails loud at the
  DB edge rather than writing silently bad data.
- **Tagged rejects for UX** — returns `"missing_team" | "missing_body"
  | "body_too_long"` so the Server Action can map each to a distinct
  flash. The validator's bare `null` is the right shape for the pass-or-
  fail seam boundary but too coarse for operator UX; the helper closes
  that gap.

Other pinned decisions:
- **Body stored AS-IS (no trim)** — matches the validator's "validate,
  don't rewrite" posture. Whitespace-only body is still rejected as
  `missing_body` (via `.trim().length === 0` check), but the raw bytes
  the operator typed go to the DB. A future export / diff surface
  doesn't want silently-scrubbed whitespace.
- **Length cap uses RAW length** — `body.length > policy.maxBodyLength`,
  not trimmed length. Mirrors the validator exactly so a caller can't
  pass the form helper and then fail the validator on a length
  mismatch.
- **teamId trimmed** — CUIDs never contain whitespace; stray whitespace
  is a caller bug safe to scrub silently.
- **Kind hard-coded to "fact"** — `MEMORY_KINDS` is a closed set of one
  today. The form has no kind selector, and the parser ignores any
  `rawKind` that slipped through via an untyped caller (there's a test
  pin for this).
- **Provenance source fields always null** — form writes have no chat
  context. `sourceSessionId` / `sourceMessageId` are reserved for the
  future `remember_fact` tool path where the dispatcher knows the chat
  ids.
- **`createdByUserId === ""` normalises to null** — matches the
  validator's provenance normalisation. Prisma FK can't be an empty
  string, and a dangling attribution is worse than no attribution.
- **Reason precedence: `missing_team` > `missing_body` > `body_too_long`**
  — mirrors the order an operator fills the form (team selector at top).
  Both empty → flash asks for team first. Whitespace-only body of 2000
  chars (both `missing_body` AND `body_too_long` would match) → flash
  says "enter the memory text" because that's the more actionable hint.
- **Fail-closed on null/undefined args** — helper returns
  `missing_team` rather than throwing. Strictest reject reason, and
  viewers see "select a team" which isn't leaking internal state.

### Auth helper renamed `decideMemoryDeleteAuth` → `decideMemoryMutateAuth`

P16-E.1 introduced the role+tenant decision helper specifically for
delete. P16-F observed that the decision tree — viewer→not_editor,
admin→allow, editor+member→allow, editor-not-member→not_member — is
identical for **any write-side mutation**. Rather than duplicate it
into `decideMemoryCreateAuth` or sprinkle the same tree across every
Server Action, the helper is renamed and shared. Both `createAction`
and `deleteAction` route through one function.

This means:

- Governance rules **can't drift** between create and delete — a
  future tightening (e.g. "admins lose cross-team write when not a
  member") lands in one file.
- If create and delete ever need to diverge (e.g. admin-only hard-
  delete vs editor-wide create), that split is cheap to add as
  dedicated wrappers delegating to the same primitives.

The 11 existing auth pins were all migrated; new tests were not
needed because the decision tree hasn't changed.

### Server Action flow — `createAction` in `src/app/memories/page.tsx`

Mirrors the `deleteAction` shape exactly:

1. **Re-authenticate** (`getCurrentUser()` → `/login` redirect).
2. **Parse form** via `parseCreateMemoryForm`. Tagged reason →
   distinct flash copy:
   - `missing_team` → "Select a team for this memory."
   - `missing_body` → "Enter the memory text before saving."
   - `body_too_long` → "Memory body is too long (max 1024 characters)."
3. **Gate role + tenant** via `decideMemoryMutateAuth`. Reject reason
   → flash copy:
   - `not_editor` → "Editor role required to save durable memory."
   - `not_member` → "Not authorised for that team."
4. **Write** via `createMemoryForTeam`, wrapped in `try/catch`. The
   catch is defensive — the form helper's output is guaranteed
   validator-compatible, so any throw is a programmer error. The
   operator still gets a clean "Could not save memory. Please try
   again." flash rather than a 500.
5. **Redirect to `/memories`** with the appropriate flash.

Parse runs **before** the auth check — telling an operator "body is
required" is more actionable than "not authorised for that team" when
the form wasn't filled out.

### UI

- **Form gated on `isEditor`** — same render gate as the Remove button.
  Viewers don't see the create form. A crafted POST is still refused
  by the Server Action's own `decideMemoryMutateAuth` check; the
  render gate is cosmetics to keep viewer pages uncluttered.
- **Team selector lists exactly the teamIds resolved for READ scope**
  — for non-admins, their memberships; for admins, all non-archived
  teams. `defaultValue` is the first team in the existing page
  ordering (membership order for non-admins, alphabetical for admins).
- **Always-show `<select>`** — even for single-team operators. Simpler
  than auto-hiding the control on 1-team cases, and keeps the form
  shape stable across operator types.
- **`Field` component + `className="field"`** for the select and
  textarea — matches the teams/users/campaigns form idiom. Earlier
  drafts used `className="input"`, which only existed in the old
  `/memories` page and `api/chat/route.ts`; `field` is the product-
  wide convention.
- **`Icon name="plus"`** for the form heading — matches
  contacts/templates/teams/campaigns "new" CTAs. An earlier draft used
  `Icon name="list"` which was inconsistent.
- **`maxLength={1024}` on the textarea** — client-side UX hint matching
  the policy's `maxBodyLength`. Server re-validates, so a client that
  bypasses the attribute still gets rejected with `body_too_long`.

### Copy

- **Intro copy role-aware:**
  - Editor: *"Use the form below to save new entries, and remove
    anything that looks stale or wrong."*
  - Viewer: *"Entries are saved and curated by editors on your team."*
- **Empty-state role-aware:**
  - Editor: *"Use the form above to save the first durable fact for
    your team."*
  - Viewer: *"No durable facts have been saved for your teams yet."*
- **File-header block updated to "P16-E / P16-F"** with a History
  section (P16-E, P16-E.1, P16-F) so the tranche evolution is readable
  from the top of the file.

## What's pinned by tests

`tests/unit/memory-create-form.test.ts` — **19 new pins** on the
parser. Each test name doubles as the invariant it guards:

- Happy path — canonical shape on pass
- `kind` is hard-coded to "fact" (no override via args)
- Trims `teamId` (CUIDs don't have whitespace)
- Does NOT trim `body` (validate, don't rewrite)
- Provenance source fields are always null
- `createdByUserId` empty string normalised to null
- Missing `teamId` returns `missing_team`
- Whitespace-only `teamId` returns `missing_team`
- Non-string `teamId` returns `missing_team`
- Missing `body` returns `missing_body`
- Whitespace-only `body` returns `missing_body`
- Non-string `body` returns `missing_body`
- Oversized `body` returns `body_too_long`
- `body` at exactly `maxBodyLength` passes
- Length check uses RAW bytes, not trimmed
- Custom policy override applies
- `missing_team` takes precedence over `missing_body`
- `missing_body` takes precedence over `body_too_long`
- null/undefined args fail-closed as `missing_team`

`tests/unit/memory-admin-auth.test.ts` — all 11 existing pins migrated
to the renamed `decideMemoryMutateAuth`. Comment block updated to
cover both create AND delete. No test body changes — same decision
tree, same branches.

## Non-changes in this slice

Explicitly NOT touched so the blast radius stays minimal:

- `src/lib/memory/server.ts` — the write seam is unchanged. The
  P16-B validator + the DB-layer tenant invariants (the `teamId`
  pin on `create`) are untouched; P16-F just adds a caller.
- `src/lib/memory/admin-query.ts` / `src/lib/memory/ui.ts` — pure
  shape and display layers remain completely unchanged.
- `src/lib/memory/validate.ts` / `src/lib/memory/policy.ts` —
  validator and policy unchanged. The form helper consumes
  `DEFAULT_MEMORY_POLICY.maxBodyLength` but doesn't reshape it.
- `src/components/Shell.tsx` — the Memories nav link still visible
  to all authenticated users; the viewer / editor distinction is
  inside the page, not in the nav.
- No schema change. No new route. No new column.
- The chat-tool dispatcher / system-prompt rendering / recall
  budget — none touched. The read-side recall path from P16-D/D.1
  is unchanged.

## Trust-scope summary after P16-F

| Capability | Viewer | Editor | Admin |
|---|---|---|---|
| See Memories page | yes | yes | yes |
| Read memories on teams they belong to | yes | yes | yes |
| Read memories across ALL teams | no | no | **yes** |
| **Create** memories on teams they belong to | **no** | **yes** | **yes** |
| **Create** memories across ALL teams | **no** | **no** | **yes** |
| Delete memories on teams they belong to | no | yes | yes |
| Delete memories across ALL teams | no | no | yes |

The two Create rows are the behavioural additions this slice
introduces. Viewers are fully read-only on the memory surface —
seeing the page, reading entries, but neither create nor delete.

Mutation parity across Create and Delete is **intentional**. Both
route through `decideMemoryMutateAuth`, so the tenant rules can't
drift. Cross-team admin writes are not an oversight — they mirror
cross-team admin reads from P16-E.

## Audit asks for GPT

1. **Write seam callsite** — `createAction` is now the first live
   caller of the P16-B write seam. Is the flow (parse → auth → write
   with try/catch → flash + redirect) the right shape, or should
   the flash copy / error mapping be structured differently?
2. **Helper rename** — `decideMemoryDeleteAuth` →
   `decideMemoryMutateAuth` because both write-side actions share
   the decision tree. Is the rename + single-helper pattern the
   right call, or would sibling `decide{Create,Delete}Auth` wrappers
   (even if they just delegate to the same primitive) be clearer
   for future audit trails?
3. **Parse-before-auth ordering** — `createAction` runs the form
   parser before the role check. My reasoning: "body is required"
   is more actionable than "not authorised for that team" when the
   form is empty. Acceptable, or should the role check run first
   so a viewer's submit never leaks parse-level field feedback?
4. **Form UX granularity** — three distinct flash reasons for parse
   (`missing_team`, `missing_body`, `body_too_long`) + two for auth
   (`not_editor`, `not_member`) = five possible refusal flashes.
   Right granularity, or should some be collapsed (e.g. both auth
   rejects to a single "Refused")?
5. **Team selector default** — pre-selects the first team in the
   page's READ scope ordering. For admins scrolling an alphabetical
   list of dozens of teams this could easily save-to-the-wrong-team.
   Should the default be "please pick" (no defaultValue, required
   selection) for admins specifically, or is the current default
   OK given the admin also sees the team name as a heading on every
   saved memory below?
6. **Client-side maxLength hint** — `maxLength={1024}` on the
   textarea matches the policy cap. If the policy ever raises the
   cap, the UI stays capped at 1024 until someone updates the
   attribute. Worth wiring from a shared constant, or is two-place
   drift on a sub-millisecond UX hint acceptable?
7. **`<select>` always-shown** — even for single-team operators.
   Simpler than auto-hiding; trade-off is a slightly-redundant
   control for single-team users. Acceptable?

## Follow-ups deliberately punted

- **Chat-tool `remember_fact` write path.** The chat path needs an
  injection defence pattern (memory body sourced from the model must
  not be prompt-injected), a confirm ceremony (like `confirm_send`
  today), and source-id provenance from the dispatcher. Dedicated
  slice.
- **In-place edit.** Delete + re-save covers the correction case
  in the current UI; a second write seam for edit would duplicate
  validator logic and isn't requested by any product need I know of.
- **Archive (soft-delete).** Requires the P16-C recall builder's
  filter update + a schema migration. P16-F specifically avoids
  touching recall behavior.
- **Pagination.** `listDefaultLimit = 50` per team is still plenty
  for the near-term operator caseload.
- **Search / filter by kind.** `MEMORY_KINDS` is a closed set of
  one today; a filter is premature.
- **Per-team maxBodyLength (premium-tier cap).** The parser already
  accepts a policy override so the knob exists; wiring it to team
  state is future work.
