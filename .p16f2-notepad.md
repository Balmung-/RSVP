## What this slice does

Closes the one non-blocking residual GPT mentioned in both the P16-F
block and the P16-F.1 green-light audits:

> The textarea `maxLength={1024}` still duplicates the policy
> constant at the render layer. Fine for now, but if the memory
> policy cap ever changes, that UI literal should be derived from
> the same source.

No user-visible behaviour change at the default policy value (the
cap is 1024 today). The slice is a pure coupling fix so the
parser and the render layer can't drift when `DEFAULT_MEMORY_POLICY.maxBodyLength`
eventually moves.

### Three literals folded into one constant

The string `"1024"` appeared in three places in
`src/app/memories/page.tsx` prior to this slice:

| Location | Role | Before | After |
|---|---|---|---|
| Server Action | `body_too_long` flash copy | `"Memory body is too long (max 1024 characters)."` | `` `Memory body is too long (max ${DEFAULT_MEMORY_POLICY.maxBodyLength} characters).` `` |
| UI attribute | textarea length cap | `maxLength={1024}` | `maxLength={DEFAULT_MEMORY_POLICY.maxBodyLength}` |
| Helper text | under the form | `"Max 1024 characters. Saved as durable context for this team."` | `` `Max ${DEFAULT_MEMORY_POLICY.maxBodyLength} characters. Saved as durable context for this team.` `` |

The parser itself (`src/lib/memory/form.ts`) already imported the
policy directly — the drift risk was entirely in the render layer.

### Comment scar intentionally kept

The file-header History block explicitly names the three places
that used to carry the literal, so a future reader can search the
file for "1024" and land on that annotation instead of thinking
"where did this couple?" This is the same discipline used in
earlier slices (e.g. the P16-C.1 invariant chain "recallMaxLimit
<= recallScanMaxLimit <= listMaxLimit" which names the numbers
in a comment so a future lookup resolves).

## Why this was worth doing vs deferring

GPT marked it non-blocking twice, so on its own it wouldn't force
a slice. The reason to fold it in now rather than punt:

- **Clean tranche boundary.** The P16 memory tranche (A → F.2) is
  the last memory work for the foreseeable future; the next queue
  items are WhatsApp document/PDF send and launch hardening,
  neither of which touches the memory surface. Leaving a
  called-out drift risk at the tranche boundary means the next
  time someone changes the policy cap, they have to go back and
  find the three render-layer literals manually.
- **One-line cost.** The whole change is 4 edits totalling 13
  inserted / 5 removed lines. The audit overhead of a dedicated
  slice (notepad + Agent chat append + GPT re-audit turn) is the
  larger part of the cost.
- **No blast radius.** No parser, no auth, no server seam, no
  tests touched. Pure render-layer constant-coupling.

## What's pinned by tests

**No new pins.** Existing coverage protects the invariant the way
constant-coupling should be protected:

- `tests/unit/memory-create-form.test.ts` → `parseCreateMemoryForm: body at exactly maxBodyLength passes` uses `"a".repeat(1024)` and `DEFAULT_MEMORY_POLICY.maxBodyLength` to compute the boundary. If the policy bumps to 2048, that test's input length will need to change AND the parser will accept the longer body; the invariant "at exactly the cap, body passes" is policy-parameterised.
- `tests/unit/memory-create-form.test.ts` → `oversized body returns 'body_too_long'` uses `"a".repeat(1025)` referencing the same `1024` comment. Same pattern — if the policy bumps, the test input must bump too.
- `tests/unit/memory-validate.test.ts` (from P16-B) uses the same pattern.

A render-layer pin ("the `<textarea>` maxLength attribute equals
the policy value") would require pulling in Testing Library /
jsdom for the Server Component; the single-constant coupling is
low enough risk for code-review + type-check to catch. If P16 ever
gets a render-layer test harness for other reasons, that pin
becomes the obvious first test.
